    Current pattern (WEAK):
    ```yaml
    rescue:
      - name: Record expected failure
        ansible.builtin.set_fact:
          windows_missing_image_failed: true
    ```


<task type="auto">
  <name>Task 3: Simplify LB Service deletion by removing redundant failed_when guard</name>
  <files>collections/ansible_collections/osac/templates/roles/ocp_virt_vm/tasks/delete_resources.yaml</files>
  <action>
    Locate the "Delete load-balancer Service" task in delete_resources.yaml. Remove the failed_when guard if it follows this pattern:

    ```yaml
    failed_when:
      - delete_lb_service.failed is defined
      - delete_lb_service.failed
      - "'not found' not in (delete_lb_service.msg | default(''))"
    ```

    Why redundant: kubernetes.core.k8s with state: absent already handles missing resources gracefully (returns changed=false when resource doesn't exist). The guard adds complexity without value.

    Keep the task simple:
    ```yaml
    - name: Delete load-balancer Service
      kubernetes.core.k8s:
        kubeconfig: "{{ remote_cluster_kubeconfig | default(omit) }}"
        api_version: v1
        kind: Service
        name: "{{ compute_instance_name }}-load-balancer"
        namespace: "{{ compute_instance_target_namespace }}"
        state: absent
      # No failed_when needed — module is idempotent
    ```

    If the failed_when pattern is different or serves a legitimate purpose (network timeout handling, permissions), document why it's needed in a comment.

    Research indicates this guard is for "not found" suppression, which is unnecessary.
  </action>
