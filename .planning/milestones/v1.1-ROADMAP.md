# Roadmap: ocp_virt_vm Consolidation

## Overview

This roadmap consolidates the separate `windows_oci_vm` template role into `ocp_virt_vm`, creating a unified Linux and Windows compute template. The consolidation reduces duplication, maintains the proven `ocp_virt_vm` pattern, and establishes OS-conditional branching as the standard for multi-OS compute templates.

## Phase Details

### Phase 1: ocp_virt_vm Consolidation

**Goal**: Unify Linux and Windows VM provisioning in a single `ocp_virt_vm` template with OS-conditional branching

**Depends on**: v1.0 (Windows VM Provisioning shipped and validated)

**Requirements**: CONS-01, CONS-02, CONS-03, CONS-04

**Success Criteria** (what must be TRUE):
1. `ocp_virt_vm` role handles both Linux and Windows VMs via `guest_os_family` branching
2. OS family is inferred from `osac.openshift.io/guest-os-family` annotation or `containerdisks/windows` image path heuristic
3. All Windows-specific tasks (sysprep hostname, Hyper-V enlightenments, clock config) executed only when `guest_os_family == 'windows'`
4. All Linux-specific tasks (cloud-init, SSH key) executed only when `guest_os_family == 'linux'`
5. `windows_oci_vm` role deleted (16 files removed, zero references remain)
6. Create/delete resource symmetry: every resource created in `create_*.yaml` has matching delete in `delete_resources.yaml`
7. `meta/osac.yaml` updated with unified template metadata
8. Argument specs documentation clarifies OS-dependent defaults (exposed_ports, Hyper-V features)
9. OSAC catalog registration uses single `template_id: osac.templates.ocp_virt_vm` for both OS families
10. Integration tests pass: Linux baseline, Windows baseline, and overrides test execute without errors

**Plans**: 3 (complete)
