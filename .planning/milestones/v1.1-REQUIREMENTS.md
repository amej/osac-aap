# Requirements: ocp_virt_vm Consolidation

## Overview

Consolidate the separate `windows_oci_vm` template role (created in v1.0) into `ocp_virt_vm`, creating a unified Linux and Windows compute template with OS-conditional branching.

## Functional Requirements

### CONS-01: OS-Conditional Template Logic

**Requirement**: `ocp_virt_vm` must handle both Linux and Windows VMs with OS-specific branching.

**Acceptance Criteria**:
- Role includes `tasks/infer_guest_os_family.yaml` that sets `guest_os_family` variable
- Inference checks `osac.openshift.io/guest-os-family` annotation on ComputeInstance
- Falls back to `containerdisks/windows` substring match in `spec.image.sourceRef`
- Defaults to `linux` if neither annotation nor image path indicate Windows
- All OS-conditional tasks use `when: guest_os_family == 'windows'` or `when: guest_os_family == 'linux'`

### CONS-02: Windows-Specific Task Migration

**Requirement**: All Windows-specific tasks from `windows_oci_vm` must be integrated into `ocp_virt_vm`.

**Acceptance Criteria**:
- Sysprep hostname configuration (15-char NetBIOS truncation) executes only for Windows
- Hyper-V enlightenments (10+ features) applied only for Windows
- Windows clock configuration (domain, timezone support) applied only for Windows
- CloudBase-Init user-data delivery via cloudInitNoCloud volume type for Windows
- Sysprep disk as SATA CD-ROM bus (required by Windows Setup)
- Windows VM ready timeout of 900s (vs 600s for Linux)
- Linux cloud-init and SSH key injection skipped for Windows

### CONS-03: Role Cleanup and Deletion

**Requirement**: The `windows_oci_vm` role must be completely removed; zero references remain in codebase.

**Acceptance Criteria**:
- `collections/ansible_collections/osac/templates/roles/windows_oci_vm/` directory deleted
- No playbook, role, or documentation references to `windows_oci_vm` exist outside of archived planning docs
- `meta/argument_specs.yaml` and `meta/osac.yaml` in `ocp_virt_vm` do not mention `windows_oci_vm`
- Integration tests do not reference the deleted role
- OSAC catalog registration uses `template_id: osac.templates.ocp_virt_vm` for both Linux and Windows

### CONS-04: Create/Delete Resource Symmetry

**Requirement**: Every resource created in `create_*.yaml` files must have a matching delete task.

**Acceptance Criteria**:
- `tasks/delete_resources.yaml` contains a delete task for every resource created
- No orphaned delete tasks (e.g., cloud-init secret deletion without corresponding create)
- Orphaned tasks from Linux-specific defaults removed if not applicable to Windows
- Create/delete pairs verified through systematic grep audit

## Non-Functional Requirements

### Performance

- Consolidation must not impact VM provisioning latency
- OS inference (annotation check, image path check) must complete in <100ms
- No new network calls or external dependencies

### Compatibility

- Must not break existing Linux VM provisioning workflows
- Must maintain backward compatibility with `ocp_virt_vm` CRD spec
- OS family inference must gracefully handle missing annotation and non-matching image paths

### Documentation

- `meta/osac.yaml` must reflect unified template with both Linux and Windows capabilities
- `meta/argument_specs.yaml` descriptions must clarify OS-dependent defaults (exposed_ports, Hyper-V features)
- README updated to reflect unified template pattern

2. Deleted `windows_oci_vm` role (16 files removed)
